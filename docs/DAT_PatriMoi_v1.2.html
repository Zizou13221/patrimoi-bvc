1 Contexte et portée

PatriMoi est une application iOS de suivi de patrimoine personnel destinée aux Marocains résidant au Maroc ou à l'étranger. Elle couvre 8 catégories d'actifs : immobilier, PEA (Bourse de Casablanca), compte-titres, or, assurance-vie, liquidités (comptes bancaires + devises), retraite, et autres.

L'application propose deux modes :

  • Mode démo — données INIT (~1,5 M DH) pré-remplies, aucun compte requis, persistance locale uniquement (MMKV)
  • Mode connecté — compte Supabase, sync cloud, snapshots historiques, abonnement PatriMoi+ (49 DH/mois)

Ce DAT décrit la version 1.2, l'état de l'art post-refactoring des sessions de juillet 2026. Il couvre l'existant déployé — pas les évolutions futures.

2 Vue d'ensemble de l'architecture

PatriMoi suit une architecture client-heavy, BaaS-backed : toute la logique métier tourne dans le client React Native ; le backend (Supabase) assure uniquement la persistance, l'authentification, et le cache des données de marché.

Diagramme C4 — Niveau contexte
Principe directeur

Le client est la source de vérité pour les données utilisateur. Supabase est la sauvegarde cloud, pas un ORM. Toutes les mutations transitent par l'outbox MMKV — Supabase reçoit une copie asynchrone.

Le choix de rester sur l'Old Architecture est intentionnel : la New Architecture (JSI/Fabric) impose une réécriture des native modules et n'apporte pas de gains utilisateur mesurables pour cette app à dominante formulaire. La migration sera envisagée en v2.0.

Hermes est le seul moteur supporté — V8 et JavaScriptCore ne font pas partie du pipeline CI. Cela impose deux contraintes :

  • AbortSignal.timeout() non disponible → makeTimeoutSignal(ms) maison avec AbortController
  • crypto.getRandomValues() disponible depuis Hermes 0.13 / RN 0.74 → fallback Math.random() inclus dans _generateKey() et _randomSalt()

3.2 Configuration deux dépôts

La base de code est organisée en deux répertoires distincts sur la machine de développement :

Setup two-repos — flux de développement

PatriMoiApp est le projet React Native créé par react-native init, avec son Podfile, son projet Xcode et ses modules natifs (react-native-mmkv, react-native-keychain, etc.) déjà liés et compilés. Le dépôt source PatriMoi contient la logique métier, les composants et la CI. Le split est un choix pragmatique : fusionner les deux nécessiterait de recréer la liaison native depuis zéro (CocoaPods, Xcode target, xcconfig). Le script apply_modifs.command maintient la synchronisation. À noter : la CI (ci.yml) ne compile aucun code natif — elle n'exécute que Jest et tsc --noEmit sur ubuntu-latest ; le build natif (release.yml) tourne sur macos-15 qui dispose d'Xcode.

Fichiers exclus de la synchronisation (resteront ceux de PatriMoiApp) :

package.json · package-lock.json · metro.config.js · babel.config.js · jest.config.js · tsconfig.json · index.js · Podfile · apply_modifs.command · bvc.yml · or.yml
Point de vigilance — Metro shadowing

Metro résout .ts avant .js. Si un fichier template auth.ts existe dans PatriMoiApp, il shadow notre auth.js et le module factory peut avorter. apply_modifs.command supprime systématiquement les .ts/.tsx conflictuels avant chaque déploiement.

4 Couche stockage

PatriMoi utilise une architecture de stockage en trois niveaux, chacun avec un rôle précis :

NiveauTechnologieContenuSécurité
Niveau 1 (primaire)MMKV patrimoi_v2Données patrimoine JSON complètesAES-128, clé en Keychain
Niveau 1 (clé)iOS KeychainClé AES-128 de MMKV + JWT Supabase + PIN hashProtégé par des clés dérivées du Secure Enclave, WHEN_UNLOCKED_THIS_DEVICE_ONLY
Niveau 2 (legacy)AsyncStorageCopie patrimoine (fallback offline + lazy migration)Non chiffré
Niveau 3 (outbox)MMKV patrimoi_v2Queue mutations, verrou, copie locale la plus récenteAES-128 (même store)

4.1 MMKV chiffré

Deux instances MMKV distinctes sont utilisées :

  • patrimoi_key (id='patrimoi_key', non chiffré) — stockage temporaire de la clé AES-128 avant migration Keychain. Vide post-migration.
  • patrimoi_v2 (id='patrimoi_v2', encryptionKey=<clé AES-128>) — toutes les données utilisateur + outbox queue.

MMKV est choisi car ses lectures sont synchrones (contrairement à AsyncStorage qui impose des await), ce qui simplifie le code des reducers Zustand et évite des race conditions lors du chargement des données au démarrage.

4.2 Keychain iOS

Trois services Keychain sont utilisés, chacun isolé par un service ID distinct :

ServiceContenuAccessible
patrimoi.mmkv.keyClé AES-128 pour MMKV patrimoi_v2WHEN_UNLOCKED_THIS_DEVICE_ONLY
patrimoi.supabase.*Session JWT Supabase (access + refresh token)WHEN_UNLOCKED_THIS_DEVICE_ONLY
patrimoi.pinPIN hashé SHA-256 (<hash>:<salt>)WHEN_UNLOCKED_THIS_DEVICE_ONLY

L'attribut WHEN_UNLOCKED_THIS_DEVICE_ONLY garantit que la clé est absente des backups iTunes non chiffrés et liée à l'appareil physique. La restauration sur un nouvel appareil repart d'une installation fraîche, avec récupération des données depuis Supabase.

4.3 Phases d'initialisation du stockage

Séquence boot — initialisation stockage (storage.js)
Fix critique v1.2 — Préservation de clé (scope bug + fallback)

Sur simulateur ou build Debug sans entitlement Keychain Sharing, KC.setGenericPassword() lance errSecMissingEntitlement (-34018). Dans le catch du step 2, le fallback if (mmkvKey) key = mmkvKey est critique : sans lui, key reste null → step 3 génère une nouvelle clé → patrimoi_v2 s'ouvre avec une clé différente → perte totale des données.

Bug de scope corrigé (juillet 2026) : mmkvKey était déclaré const à l'intérieur du bloc try {} — une variable const/let de bloc try n'est pas accessible dans le catch correspondant. typeof mmkvKey retournait 'undefined' (pas de ReferenceError, mais condition toujours fausse), rendant le fallback silencieusement inopérant. Corrigé en hoissant la déclaration (let mmkvKey = null) avant le try.

Sentry : ce cas d'usage émet l'événement keychain_unavailable_fallback (filtre entitlement : silencieux sur simulateur, actif sur device sans Keychain Sharing configuré).

Step 5 — Migration PIN au boot (v1.2.0) — one-shot

Ajouté dans initStorage() après le step 4. Conditionné par le flag pin_migration_v2 dans _store (MMKV patrimoi_v2) : si le flag est absent, le step s'exécute — lit patrimoi.pin via Keychain, vérifie via isPinLegacy(), et si l'ancien format est détecté appelle hashPin() pour réécrire en <sha256>:<salt>. Le flag est ensuite écrit ('1') que le PIN existe ou non, court-circuitant définitivement le step aux boots suivants.

Pourquoi le flag : un appel KC.getGenericPassword() coûte 50–300 ms selon l'état du device. Sans guard, le step 5 ajoutait ce délai à chaque démarrage. Avec le flag, le surcoût n'existe qu'au premier boot post-v1.2.0. Le boot est le seul moment où la migration est possible sans redemander le PIN à l'utilisateur. verifyPin() et isPinLegacy() continuent de gérer les deux formats pendant la transition.

5.2 Outbox pattern (syncQueue.js)

Toutes les mutations utilisateur transitent par un outbox MMKV avant d'atteindre Supabase. Ce pattern garantit la durabilité des mutations même en cas de coupure réseau ou de crash de l'application.

Flux mutation utilisateur → Supabase

Caractéristiques clés de l'outbox :

  • Coalescing — une seule entrée par userId dans la file (last-write-wins), évite les conflits multi-mutations rapides
  • Debounce 800 ms + worker toutes les 5 s — équilibre réactivité et économie de bande passante
  • Max 5 retries avec backoff exponentiel (base 1000 ms)
  • Mutations échouées persistées dans outbox_failed_v2 (max 20) + alerte Sentry
  • Verrou optimiste (M008) — RPC save_patrimoine_data détecte les conflits multi-appareils et répond had_conflict: true sans bloquer l'écriture

5.3 APIs de données de marché

SourceDonnéeTransportCache
Bourse de Casablanca (BVC)Cours actions marocainesEdge Function market-data-proxyMMKV 30 min / stale 24 h
API OrPrix or (DH/gramme)Edge Function market-data-proxyMMKV 30 min
API DevisesTaux EUR/USD/GBP/CHF → MADEdge Function market-data-proxyMMKV 30 min

Ces données sont chargées via fetchWithRetry() (3 tentatives, backoff exponentiel). Elles sont stockées dans MMKV mais jamais envoyées à Supabase (R8 — strip avant upsert). En cas d'indisponibilité réseau, l'app affiche les dernières valeurs cachées.

5.4 Sentry (monitoring)

Sentry est intégré via un module stub défensif : si le package @sentry/react-native est absent (CI, build sans DSN), toutes les fonctions deviennent des no-ops silencieux. Deux fonctionnalités de protection :

  • Scrubbing PII — les montants (/montant|valeur|total|prix/i) et les emails sont retirés avant envoi
  • Silence entitlement — _isEntitlementError() filtre les erreurs errSecMissingEntitlement (simulateur), évitant de polluer le tableau de bord Sentry

8 Conventions de développement

8.1 Zero Static Imports pattern

Tous les modules natifs sont chargés via require() dans un try/catch, jamais via des import ES6 statiques. Cette convention est notée dans les commentaires de fichier avec la mention "IMPORTANT : aucun import statique".

// ✅ Correct — Zero Static Imports
let KC = null;
try {
  const K = require('react-native-keychain');
  if (K?.getGenericPassword) KC = K;
} catch {}

// ❌ Incorrect — import statique
import * as Keychain from 'react-native-keychain'; // crashe en CI

Raison : avec Babel CJS transform (Old Architecture), les import ES6 sont compilés en require() qui s'exécutent lors de l'évaluation initiale du module factory. Si un module natif manque (CI, pod non installé), toute la factory avorte et tous ses exports restent undefined, propageant l'erreur en cascade. Le pattern require() dans try/catch isole chaque dépendance et garantit que le module factory se termine toujours.

8.3 setData vs setDataRaw — règle absolue

  • setData(fn) — toute modification des données par l'utilisateur. Déclenche la sync cloud.
  • setDataRaw(fn) — chargement de données depuis Supabase ou données de marché (cours, taux). Ne déclenche pas la sync (évite les boucles write→sync→write).

8.4 Convention de nommage des fichiers

Les fichiers source PatriMoi sont en .js ou .jsx (jamais .ts/.tsx sauf AppNavigator.tsx) pour éviter le shadowing Metro. Si un fichier .ts du template RN porte le même nom, apply_modifs.command le supprime explicitement avant la copie.

Exception — fichiers de test : les fichiers dans __tests__/ peuvent utiliser l'extension .ts (convention Jest/TypeScript — ex. migrations.test.ts, store.test.ts). Metro ne bundle pas le dossier __tests__/, il n'est donc pas soumis à la règle de shadowing.

PIN hashing — architecture de défense en profondeur

Un PIN 6 chiffres stocké en clair dans le Keychain serait lisible via un dump Keychain sur appareil jailbreaké. Le module pinHash.js implémente SHA-256 pur JS (zéro dépendance native, compatible Hermes) avec sel aléatoire crypto.getRandomValues(). Format stocké : <sha256_hex>:<salt_hex> (97 caractères).

Migration au boot (v1.2.0) : les devices ayant un PIN en ancien format (texte brut, sans :) sont migrés au premier démarrage post-v1.2.0 dans initStorage() step 5. Le step est gardé par un flag pin_migration_v2 dans MMKV : il ne s'exécute qu'une seule fois, évitant un appel Keychain superflu (50–300 ms) à chaque boot ultérieur. isPinLegacy(stored) détecte le format ancien ; hashPin(pin) produit le hash avec sel ; la valeur hachée est réécrite dans patrimoi.pin. verifyPin() gère les deux formats pendant la période de transition.

12 Argumentaire des choix techniques

12.1 React Native vs alternatives

✓ Choix — React Native

Partage de logique métier (calculs, migrations, outbox) entre iOS et un futur client Android. Écosystème mature, Hermes rapide. Permet d'itérer sur la logique sans recompiler en Swift.

Alternatives écartées

Swift natif — plus performant mais pas de partage de code avec Android futur. Flutter — excellent mais l'équipe est JS-native, et l'accès Keychain/MMKV est plus limité.

12.2 MMKV vs AsyncStorage

✓ Choix — MMKV

Lectures synchrones → pas de await dans les reducers Zustand. Chiffrement AES-128 natif. 10× plus rapide qu'AsyncStorage sur les benchmarks React Native. API simple (getString/set/delete).

AsyncStorage conservé

AsyncStorage reste comme couche legacy pour la migration progressive (smartRead : MMKV first, fallback AsyncStorage). Sera retiré quand tous les utilisateurs auront migré.

12.3 Keychain pour la clé MMKV

✓ Choix — Keychain

La clé AES-128 dans MMKV était accessible via un backup iTunes non chiffré. Le Keychain avec WHEN_UNLOCKED_THIS_DEVICE_ONLY lie la clé à l'appareil physique et l'exclut des backups. Un dump MMKV sans la clé est inutilisable.

Migration graduelle

Les utilisateurs existants (clé dans MMKV) migrent automatiquement à la première ouverture post-v1.2, sans interruption. La Phase 2 async de initStorage() gère cette transition.

12.4 Supabase vs Firebase

✓ Choix — Supabase

PostgreSQL + RLS = sécurité forte et familière. Edge Functions Deno pour le proxy de marché. Région West EU (RGPD). Open-source = pas de vendor lock-in. Coût prévisible à l'usage.

Firebase écarté

Firestore est NoSQL — inadapté à un schéma patrimoine évolutif. Les règles de sécurité Firebase sont moins expressives que RLS PostgreSQL pour l'isolation par user_id.

12.5 Zustand vs Redux

✓ Choix — Zustand

API minimaliste (20 lignes de store), pas de boilerplate action/reducer/selector. Sélecteurs natifs (usePatrimoineStore(s => s.field)) sans connect() HOC. Bundle size minimal.

Redux écarté

Redux Toolkit aurait imposé des slices, des middlewares, et une configuration RTK Query — overhead disproportionné pour un store mono-entité. La distinction setData/setDataRaw est plus lisible en Zustand.

12.6 Navigation maison vs React Navigation

✓ Choix — routeur Zustand

L'app a une navigation tabulaire simple sans deep linking ni stack imbriqué. React Navigation ajouterait ~800 Ko de bundle et une complexité inutile (navigateurs imbriqués, refs, gestureHandler).

React Navigation pour la v2

Si des fonctionnalités de deep linking (notifications push → page spécifique) ou un back-stack natif sont nécessaires, la migration vers React Navigation sera envisagée.

12.7 Outbox pattern vs appel direct Supabase

✓ Choix — Outbox MMKV

Un appel direct Supabase dans setData perdrait les mutations en cas de coupure réseau ou crash. L'outbox garantit la durabilité : même si l'app est tuée entre une modification et la prochaine flush, la mutation sera envoyée au prochain démarrage.

Compromis — latence de sync

La sync peut arriver jusqu'à 800 ms après la modification (debounce). Acceptable pour une app de patrimoine personnel où la cohérence éventuelle est suffisante et la cohérence temps-réel n'est pas requise.

12.8 SHA-256 pur JS vs SubtleCrypto

✓ Choix — SHA-256 pur JS

SubtleCrypto.digest() est asynchrone et non disponible sur tous les contextes Hermes 0.13. L'implémentation pur JS (domaine public, Geraint Luff 2012) est synchrone, zéro dépendance, et suffisante pour hacher un PIN 6 chiffres.

Limite acceptée

SHA-256 pur JS est ~100× plus lent que SubtleCrypto natif, mais pour un usage ponctuel (vérification PIN), cette latence (~5 ms) est imperceptible. Le sel aléatoire protège contre les rainbow tables.
